Accurate XOR

Approach

Problem link

This problem uses a property of XOR. When XORing multiple consecutive 0s or 1s, only an odd number of 1s makes the result 1.

If there are an even number of 1s, every 1 can always be paired with another 1 so that their XOR becomes 0. The occurrence of 0 does not affect the final result, so if there are an even number of 1s, the final result is always 0.

The Xor-value of a node is defined as the bitwise XOR of all the binary values present in the subtree of that node.

This sentence in the statement says that the XOR value of a tree is the XOR sum of every node under that tree.

In other words, let the current tree’s root be rrr, and let rrr have xxx child nodes (including indirect children, such as children in its subtrees), whose values are c1∼cxc_1 \sim c_xc1​∼cx​. Then the XOR value of rrr is:

XOR⁡(r)=c1⊕c2…⊕cx−1⊕cx\operatorname{XOR}(r) = c_1 \oplus c_2 \ldots \oplus c_{x - 1} \oplus c_x XOR(r)=c1​⊕c2​…⊕cx−1​⊕cx​

Because every child node has a value of either 1 or 0, the property above tells us that if the current tree’s XOR value is 1, then an odd number of nodes in all its subtrees have value 1, and vice versa.

That is, if the XOR value of tree rrr is 1:

∑x=1ncx mod 2=1\sum_{x=1}^{n}c_x \bmod 2 = 1 x=1∑n​cx​mod2=1

The problem requires kkk subtrees to have an XOR value of 1. Therefore, for every child node in these kkk subtrees, the sum of their values must be odd.

Let odcnti\text{odcnt}_iodcnti​ be the number of child nodes with value 1 in tree iii. Let the current tree be rrr, and suppose we still need klklkl trees to have an XOR value of 1 (that is, some trees already have an XOR value of 1).

If kl>0kl > 0kl>0 and odcntr mod 2=0\text{odcnt}_r \bmod 2 = 0odcntr​mod2=0, meaning that an even number of its child nodes have value 1, we should set the value of this node to 1.

This is because kl>0kl > 0kl>0, so we need more trees with XOR value 1. Since this tree has an even number of child nodes with value 1, its XOR value is not 1. Changing the value of the tree itself to 1 changes its XOR value to 1, achieving our goal.

Conversely, if kl=0kl = 0kl=0, we do not need more trees with XOR value 1. However, if odcntr mod 2=1\text{odcnt}_r \bmod 2 = 1odcntr​mod2=1, meaning the sum of the values of all its child nodes is odd, we should set rrr to 1.

This is because we do not want to produce more trees with XOR value 1. Setting rrr to 1 makes the sum of the values of all its nodes even, and the XOR value of rrr becomes 000.

With these two conclusions, we can use DFS to find the answer.

Strict Permutation

Approach

Problem link

My original idea was to sort every constraint by position, and then by value if the positions were equal.

Then I would traverse every constraint and alternately insert each constraint and each unrestricted value (according to their values, because the problem asks for the lexicographically smallest result). The explanation here is probably unclear; the following was my previous code:

This reckless approach causes a problem. Suppose we sort the constraints as described above and call them c1∼mc_{1 \sim m}c1∼m​.

Then the number in cic_ici​ can only appear in the interval (ci−1,ci](c_{i - 1}, c_i](ci−1​,ci​], which does not satisfy the problem’s requirements. This is why it received so many WAs.

The correct solution is to calculate from back to front.

We maintain a max-heap pqpqpq and traverse every position (the positions in the permutation) from back to front.

If the position of some constraint is the current position, we add the value of that constraint to pqpqpq. For every position traversed, we can then directly take the top element from pqpqpq and put it into the answer.

Therefore, we can take the value of a constraint from pqpqpq only when the current position is smaller than the constraint’s position, so every element taken from pqpqpq is valid.

At the same time, these elements are also the largest possible. Since we traverse from back to front, this ensures that the resulting permutation is lexicographically smallest.

Finally, we need to consider when to output -1. Since pqpqpq stores every element that is valid for the current position, if nothing can be taken from pqpqpq, then no valid permutation can be produced.

One final point: for numbers without any constraints, we can add them to pqpqpq at the beginning, or equivalently, their constraint position is nnn.
